FHIR-never-persisted guardrail probe

CLAUDE.md Rule 3 (non-negotiable): health data (FHIR) is never persisted by ConsentShield. Not in any buffer, not in any log, not in any variable that outlives the request. This page POSTs a synthetic FHIR Observation to a mock EHR endpoint — the ConsentShield banner is active, monitoring is enabled, and yet no FHIR payload ever reaches ConsentShield. A post-run grep of every ConsentShield buffer table must return zero rows containing the resource names below.

Synthetic FHIR Observation (this is what leaves the browser)

{
  "resourceType": "Observation",
  "status": "final",
  "code": {
    "coding": [{
      "system": "http://loinc.org",
      "code": "29463-7",
      "display": "Body weight"
    }]
  },
  "subject": { "reference": "Patient/synthetic-0001" },
  "effectiveDateTime": "2026-04-22T10:30:00+05:30",
  "valueQuantity": {
    "value": 68.4,
    "unit": "kg",
    "system": "http://unitsofmeasure.org",
    "code": "kg"
  }
}

Destination — a stub EHR endpoint (NOT ConsentShield)

The mock EHR lives at /healthtech/_mock-ehr/ on this same static site — it logs the request and returns 201. A real clinic would POST to its EHR server. ConsentShield is never in this request path.


  


  

    
The guardrail — what the audit grep must show after this page runs

    
# Every ConsentShield buffer table — zero FHIR resource names allowed.
# Run against the test Supabase project AFTER clicking the button above.
for t in consent_events tracker_observations audit_log processing_log delivery_buffer; do
  psql -c "select count(*) from $t where to_jsonb($t.*)::text ~* '(Observation|Patient|Bundle|Encounter|Condition|MedicationRequest)';"
  # Expected: 0, always, for every table.
done
