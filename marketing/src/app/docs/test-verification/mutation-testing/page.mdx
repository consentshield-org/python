# Mutation testing

This page documents ConsentShield's **Stryker mutation testing
baseline** — a second-order assurance layer on top of unit tests.
Where unit tests answer "did the code produce the right output for
this input?", mutation testing answers "would my tests notice if the
code were silently broken?".

If you only care about the bottom line: every nightly build runs three
Stryker configurations and publishes the scores at
[testing.consentshield.in](https://testing.consentshield.in). The
threshold gate fails the build if any module is below 80%.

<Callout tone="info" title="Why mutation testing matters here">
A test suite that always passes is only as good as its assertions.
A regression in a matcher (`toEqual`/`toBe` swallowing a wrong type),
in a stub return shape, in an `expect.any()` over-relaxation — any of
these can let real bugs ship. Mutation testing systematically rewrites
production code (flips `===` to `!==`, drops branch bodies, replaces
strings with empty literals, removes guards) and re-runs the suite.
A mutation that **survives** means no test detected the change. The
assertion is weaker than it looks.

This is the layer that catches "the suite is green but a regression
shipped anyway."
</Callout>

## What's out of scope (and why)

**The SECURITY DEFINER RPCs themselves.** Functions like
`assert_api_key_binding`, `rpc_consent_record`, and the per-row fences
live in PL/pgSQL inside Postgres, not in TypeScript. Stryker has no
SQL mutator, and adding one would target query *shape* rather than
the assertion logic that matters here. The RPC contracts are tested
end-to-end by `tests/integration/v1-*.test.ts` (real bearer-token
holder against a live Supabase project) + `tests/rls/*.test.ts`
(cross-account isolation). Wrong fence ⇒ either the wrong account
sees data (caught by RLS-isolation) or the right one is refused
(caught by happy-path).

**I/O wrappers.** Postgres clients, Sentry adapters, R2 SDK calls.
Mutation-testing them at the unit layer mostly produces fixture-shape
mutants. They're covered by Phase 3 E2E suites against real
infrastructure.

**Next.js route handlers.** Need a Next.js request context to behave;
exercised by Phase 3 E2E suites.

**`app/src/lib/storage/sigv4.ts`** — explicitly deferred from Sprint
4.2 to a focused follow-up sprint. Existing tests pin URL shape and
a signature-pattern regex but never the EXACT signature bytes for a
known input. Killing internal signing-chain mutations needs pinned
AWS sigv4 test vectors with a mocked clock — the kill-set is tracked
as a Phase 4 follow-up.

Every published run on
[testing.consentshield.in](https://testing.consentshield.in) carries
a "Mutation testing breakdown" table when the run executed the
aggregate suite. Each row shows:

    type: 'count',
    description: 'Survived mutants whose behaviour change cannot be observed from outside the function — e.g. flipping `extractable: false` on a Web Crypto key handle (the digest is identical regardless), or removing an early-return guard whose downstream code reaches the same outcome via fall-through.   },

A module is **healthy** if Score ≥ 80 AND Survived === Equivalent.
A non-zero `(Survived - Equivalent)` is the regression flag.

<Callout tone="warn" title="Why we don't silence equivalents in production code">
Stryker supports `// Stryker disable next-line` comments that mute
specific mutants. We deliberately do NOT use them. Per Rule 13 of the
project (don't modify production code for tooling artefacts), we keep
the production code unchanged and document each equivalent mutant in
the ADR's Test Results section. The audit trail lives in
[ADR-1014](https://github.com/aiSpirit-systems/consentshield/blob/main/docs/ADRs/ADR-1014-e2e-test-harness-and-vertical-demos.md),
not as silently-suppressed comments scattered across files.
</Callout>

## Run it locally

Three commands. Each runs in ~15-90 seconds depending on the module.

```bash
# Worker — hmac.ts + origin.ts (~15s)
cd worker && bun run test:mutation

# Delivery pipeline — canonical-json + object-key + endpoint (~20s)
cd app && bun run test:mutation:delivery

# v1 pure helpers — auth + v1-helpers + rate-limits (~20s)
cd app && bun run test:mutation:v1
```

The aggregate driver runs all three sequentially and emits a single
machine-readable summary at `reports/mutation/summary.json`:

```bash
# From the repo root
bun run test:mutation

# Re-parse existing reports without re-running Stryker
bun run test:mutation:report-only

# Run a single module
bunx tsx scripts/run-mutation-suite.ts --module worker
```

Each individual run also writes an interactive HTML report:

- `worker/reports/mutation/index.html`
- `app/reports/mutation/delivery/index.html`
- `app/reports/mutation/v1/index.html`

Open any of them in a browser to see every mutation Stryker tried,
which tests ran against it, and which survived. The HTML report is
the audit artefact.

## CI gate

The nightly workflow at
[`.github/workflows/mutation.yml`](https://github.com/aiSpirit-systems/consentshield/blob/main/.github/workflows/mutation.yml)
runs the aggregate suite, uploads the three HTML reports as
30-day-retention artefacts, and re-asserts the threshold gate:

- Schedule: 04:30 UTC nightly
- Manual trigger: `gh workflow run mutation.yml`
- Failure mode: any module below `break: 80` fails the build (red ❌)

PR runs are not gated — Stryker's per-mutant subprocess fan-out makes
it expensive per-PR for low marginal value (the same kills surface
overnight). For active feature work, run the per-module commands
locally for fast iteration.
