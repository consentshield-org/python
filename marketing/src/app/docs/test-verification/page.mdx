# Reproduce our tests

This page is for reviewers — audit firms, BFSI prospects, enterprise
procurement, compliance officers — who need to **independently verify**
that the ConsentShield end-to-end test suite does what we say it does,
against state you control.

You will: clone the repo, point it at your own Supabase test project,
seed three fixtures (ecommerce, healthcare, BFSI), run the suite, and
verify the sealed evidence archive. Wall-clock budget: **~30 minutes**
on a warm machine with a Supabase project already created.

<Callout tone="info" title="Why we built this">
Every auditor-facing test-coverage claim we make must be re-runnable
against state the reviewer controls. A shared read-only sandbox is not
sufficient evidence — you cannot trust state you did not write.
This page is the contract: everything it describes is the same code
path our own CI exercises. Tracked under
[ADR-1014](https://github.com/aiSpirit-systems/consentshield/blob/main/docs/ADRs/ADR-1014-e2e-test-harness-and-vertical-demos.md).
</Callout>

## What you get

- **Three vertical-specific fixtures** — one account each for an
  ecommerce retailer, a clinic, and a BFSI fintech. Each fixture owns
  an organisation, three web properties (live + localhost probe +
  sandbox), a consent banner tuned for its vertical, and a sandbox API
  key.
- **A full pipeline** — the browser-driven suite exercises banner →
  Cloudflare Worker HMAC → buffer → delivery → R2 object → receipt.
  Negative controls run alongside positives; mutation testing gates
  the whole run.
- **A sealed evidence archive** — every run writes a content-hashed
  bundle of Playwright reports, response bodies, and per-test trace
  IDs. A tiny verifier CLI re-hashes the bundle and refuses any
  tampering.

## Four-step bootstrap

### 1. Clone + install

```bash
git clone https://github.com/aiSpirit-systems/consentshield.git
cd consentshield
bun install
```

One checkout, one `bun install`. The workspace pulls in every app
(`app/`, `admin/`, `marketing/`, `worker/`, `tests/e2e/`) and the
shared packages.

### 2. Apply migrations against your Supabase project

```bash
export SUPABASE_ACCESS_TOKEN=sbp_...       # from supabase.com → account → access tokens
export SUPABASE_PROJECT_REF=abcdefghij      # your project's ref
bunx supabase link --project-ref "$SUPABASE_PROJECT_REF"
bunx supabase db push
```

Migrations are ordered and idempotent against an empty project. Expect
~90 seconds; subsequent re-runs are a no-op. If anything fails, scroll
back to the first error — later failures cascade from it.

### 3. Run the partner bootstrap

```bash
bunx tsx scripts/partner-bootstrap.ts
```

You will be prompted for four values:

On completion, the script writes `.env.partner` at the repo root with
mode `0600`. The file is gitignored. It contains every fixture ID
(accounts, orgs, web_properties, banners, api_keys), their signing
secrets, and a plaintext sandbox `cs_test_*` Bearer key per vertical.

Re-run with `--force` to wipe and rebuild every fixture:

```bash
bunx tsx scripts/partner-bootstrap.ts --force
```

### 4. Run the suite

```bash
cd tests/e2e && bun run install:browsers && cd ../..
bun run test:e2e:partner
```

`test:e2e:partner` is a thin alias that sets `PLAYWRIGHT_PARTNER=1` so
the harness loads `.env.partner` rather than our internal `.env.e2e`.
A full chromium + webkit pass takes roughly 5–8 minutes depending on
network round-trip to your Supabase region. Pipeline tests that
require a Worker skip with a clear message if `WORKER_URL` is unset —
RPC-level contract tests still run and still produce evidence.

## Verify the sealed evidence archive

Every run writes an archive to `tests/e2e/evidence/<commitShort>/<runId>/`:

```
manifest.json        run metadata + per-test outcomes + attachment inventory
seal.txt             SHA-256 ledger + root hash over the whole archive
attachments/
  playwright-report/ HTML report (open in browser)
  results.json       Playwright JSON reporter output
  responses/*.json   raw HTTP response bodies from each test
  trace-ids/*.txt    per-test trace IDs
```

Verify integrity with the bundled CLI:

```bash
bunx tsx scripts/e2e-verify-evidence.ts tests/e2e/evidence/<commit>/<runId>
```

Partner runs are expected to match our reference run on **per-test
pass/fail outcomes** but will differ on:

- Timings — round-trip to your Supabase region is not the same as ours.
- Trace IDs — ULID-shaped, one per test, regenerated every run.
- Response payload UUIDs — fixture orgs/accounts/properties have
  fresh IDs on every `--force` rebuild.

Outcome shape is what matters. Any per-test `status: 'failed'` in
your `manifest.json` that is `'passed'` in ours is a divergence worth
emailing us about.

## Sacrificial "must-fail" controls

Eight intentionally-broken tests run alongside the positives to prove
the harness's assertion layer is still discriminating what it claims
to. Every control asserts a patently-false proposition wrapped with
Playwright's `test.fail()` inversion — a healthy run reports all
eight as **passed** overall; a rogue run fails CI.

```bash
bun run test:e2e:controls
```

Full detail — the matrix, the CI gate, how to read a red result —
lives at **[/docs/test-verification/controls](/docs/test-verification/controls)**.

## FAQ

**The bootstrap says "schema check failed — run bunx supabase db push first".** Your project's schema is empty. Re-run step 2.

**"Does not look like a Supabase key".** The validator rejects values that are neither a JWT (`eyJ...`) nor an `sb_secret_*` / `sb_publishable_*` token. Double-check you copied the whole string without line breaks.

**The browser-based pipeline tests skip with "WORKER_URL not set".** Expected unless you are running a Cloudflare Worker locally. Follow [tests/e2e/README.md](https://github.com/aiSpirit-systems/consentshield/blob/main/tests/e2e/README.md) section "Running pipeline tests against the Worker". RPC-level contract tests still run and still produce observable-state assertions.

**Can I re-run without wiping?** Yes — the bootstrap is idempotent without `--force`. It reuses existing fixtures matched by name (e.g. `e2e-fixture-ecommerce`) and only mints a new API key if no active one exists.

**Can I target a production Supabase project?** No. The bootstrap wipes fixtures with the names listed above. If those names collide with real data in your production project, that data will be deleted. Use a disposable project.

**My Supabase project is in a different region from yours. Is that a problem?** No. Latency differs; outcomes do not.
