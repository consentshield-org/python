# Rotate an API key safely

## Problem

You need to rotate a production API key — scheduled quarterly rotation,
suspected leak, departing employee, or secret-scanner flagged the key
in a public repo. Simply revoking-and-reissuing causes a traffic blip
while your fleet refreshes. Done carelessly, that blip is hours of
failed API calls and a support incident.

Zero-downtime rotation is a 4-step dance: **issue → overlap → drain →
revoke**. The 410 Gone tombstone (ADR-1011) protects you from silently
shipping a stale key to production.

<Callout tone="tip" title="Why 410 Gone, not 401">
`401 Unauthorized` is the response for keys that never existed, typos,
and malformed headers. Treating rotated keys as `401` means a stale
deployment looks indistinguishable from an attacker probe — and pages
security. `410 Gone` (with `X-Consentshield-Key-Rotated-At` +
`X-Consentshield-Key-Prefix` response headers) is a specific, actionable
signal: *"somebody rotated this key — update your deployment."* See
[key rotation & tombstones](/docs/concepts/key-rotation-and-tombstones).
</Callout>

## Incident-response variant — key definitely leaked

The clean-rotation playbook assumes the old key is *safe to keep live*
during the overlap. If the key **is known to be leaked** (found in a
public repo, found in a log, tipped off by GitHub Secret Scanning),
skip the drain:

1. **Issue the new key immediately.**
2. **Revoke the old key right away** — don't wait for drain. Pods that depended on the old key will fail; that's the correct trade-off.
3. **Deploy the new plaintext in parallel.**
4. **Accept the blip.** 410 Gone responses during the gap are loud, explicit, and page the right people.

Exposed-secret rotations are intentionally disruptive — the cost of a
few failed API calls is trivial against the cost of a leaked key sitting
valid for another hour.

## What to monitor

- **`last_used_at` drift** — during overlap, both keys' `last_used_at` should advance. If only the old key moves, the new key isn't deployed; if only the new key moves, drain is done.
- **`rate_tier` + `scopes`** — verify the new key inherited the exact same scopes. Rotation from the dashboard preserves scopes; an operator manually re-creating a key might not.
- **410 count** — post-rotation, if your integration's error monitor shows 410s, *something is still carrying the old plaintext*. Find and fix it; don't suppress the alert.
- **401 count** — should drop to baseline. A sudden 401 spike means your deploy went wrong (typo in the new plaintext).

## Gotchas

### Secret-scanner-triggered rotation paths

GitHub Secret Scanning and similar tools often open a PR or post a
webhook when a `cs_live_*` key leaks. If your repo is public, assume
the key is compromised within minutes of the commit being pushed —
not hours. Go straight to the incident-response variant above.

### Don't rotate and revoke in one admin action

The dashboard's "Rotate" button issues a new key **and** revokes the
old one after a grace period (default: 24 h). If you want overlap
longer than 24 h, click "New key" separately and manage the old key's
lifecycle yourself via "Revoke" when drain is done.

### Scopes on the new key

A rotated key inherits the old key's scopes by default. If you're
narrowing scope as part of the rotation (e.g. dropping `admin:keys`
that no production code needs), do the deploy first, confirm no 403s
for 24 h, then revoke the old key. The order matters: rotating + tightening scope in one step means a traffic blip on any surface that legitimately needed the removed scope.

### Client libraries + `process.env` caching

Some server frameworks read env vars once at boot and cache them. If
your secret manager pushes a new value but your process doesn't
reload, the pod keeps using the old plaintext until restart. SIGHUP
your workers, or use a secret manager that triggers a rolling restart
on rotation.

### CI/CD secrets lag

GitHub Actions, CircleCI, etc. read the secret at workflow start. A
long-running workflow launched before the rotation will finish with
the old key. If the old key is revoked mid-workflow, late steps hit
410 Gone. Schedule rotations after deploys, not during.

### The tombstone lasts 2 years

The revoked plaintext continues to return **410 Gone** for 2 years
after revoke. That's not an accident — DPB audits can arrive late, and
proving "this access was rejected" six months later is table stakes.
Don't re-issue the same plaintext "because it expired from the cache";
it hasn't.
