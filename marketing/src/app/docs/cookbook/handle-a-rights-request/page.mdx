# Handle a rights request end-to-end

## Problem

A customer emails your support desk asking for all their data to be
erased (DPDP §13). Or calls your branch demanding access to everything
on file (§12). Or submits a correction through your mobile app (§12
too). You have **30 days** to complete the request, verify the
requestor's identity first, and leave an audit trail that survives DPB
scrutiny.

Most integrations capture these through the hosted rights portal — but
when the request lands through a branch, kiosk, call-centre, or your
own mobile app, you need the **programmatic API** so the audit trail
stays intact.

## Shape

The end-to-end flow is four steps:

1. **Verify the requestor's identity** *on your side* — internal KYC, branch-officer badge check, existing authenticated session, etc. This happens before the API call; you pass a free-text attestation.
2. **`POST /v1/rights/requests`** — captures the request with `status=new`. The response includes `sla_deadline` (30 days from creation by default).
3. **Fulfil it** — for erasure, call `POST /v1/deletion/trigger` for every property the principal touched. For access, package the data and email a download link. For correction, update your own systems and echo the change.
4. **Close the loop** — move the rights_request row to `completed` via the dashboard or internal RPC. The request becomes immutable after completion.

<Callout tone="security" title="Identity verification is on you">
The API does not re-verify the requestor's identity. The
`identity_verified_by` free-text string is a **statement of record** —
it goes into the audit trail exactly as provided and becomes the
forensic handle a DPB investigator will follow. Use stable, explicable
references: `"internal_kyc_check:2026-04-24"`, `"branch_officer:emp_4212"`,
`"existing_session_uid_abc123:mfa_enrolled"`.
</Callout>

const CS_API = 'https://api.consentshield.in/v1'

interface IntakeRequest {
  type:              'erasure' | 'access' | 'correction' | 'nomination'
  requestorName:     string
  requestorEmail:    string
  details?:          string
  channel:           'email' | 'call_center' | 'branch' | 'kiosk' | 'mobile_app' | 'other'
  identityAttestation: string     // what you verified, not the secret
  propertyId:        string       // your web property
}

export async function createRightsRequest(job: IntakeRequest) {
  const res = await fetch(\`\${CS_API}/rights/requests\`, {
    method: 'POST',
    headers: {
      Authorization:  \`Bearer \${process.env.CS_API_KEY}\`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      type:                 job.type,
      requestor_name:       job.requestorName,
      requestor_email:      job.requestorEmail.toLowerCase().trim(),
      request_details:      job.details ?? null,
      identity_verified_by: job.identityAttestation,
      captured_via:         job.channel,
    }),
  })
  if (!res.ok) throw new Error(\`rights_create_failed: \${res.status} \${await res.text()}\`)
  const { id, sla_deadline } = await res.json()
  return { id, slaDeadline: new Date(sla_deadline) }
}

export async function fulfilErasure(
  rightsId: string,
  email: string,
  propertyIds: string[],
  operator: string,
) {
  // Fire deletion triggers in parallel across every property the principal touched.
  const results = await Promise.all(
    propertyIds.map((propertyId) =>
      fetch(\`\${CS_API}/deletion/trigger\`, {
        method: 'POST',
        headers: {
          Authorization:  \`Bearer \${process.env.CS_API_KEY}\`,
          'Content-Type': 'application/json',
        },
        body: JSON.stringify({
          property_id:               propertyId,
          data_principal_identifier: email,
          identifier_type:           'email',
          reason:     'rights_request',
          actor_type: 'operator',
          actor_ref:  operator,
        }),
      }).then(async (r) => ({ propertyId, status: r.status, body: await r.json() })),
    ),
  )
  const failures = results.filter((r) => r.status !== 202)
  if (failures.length) throw new Error(\`partial_fulfilment: \${JSON.stringify(failures)}\`)
  return results.map((r) => ({
    property_id:  r.propertyId,
    revoked:      r.body.revoked_count,
    receipt_ids:  r.body.revoked_artefact_ids,
  }))
}

    if (!pending.length) return items

  }

## UX — what the requestor expects

- **Acknowledgement within 48 hours** of request creation. DPDP §11 doesn't spell out "acknowledgement" separately from "completion" but audit-ready providers send one anyway.
- **Status-at-a-glance** page for the requestor — they can check without contacting support.
- **Closure email on completion** with the rights-request `id` + completion timestamp + summary of what was done.
- **Escalation path** surfaced on every communication (DPB complaint route + your own grievance officer).

The hosted `/rights` portal ships all four out of the box. When you
use the programmatic API, you own this UX.

## Gotchas

### Email is lower-cased + trimmed server-side

`requestor_email` is normalised before storage (`toLowerCase().trim()`
equivalent). Do the same on your side if you correlate the request
with other records in your own database.

### `captured_via=api` is the default

If you pass `captured_via=api` or omit the field, the audit trail
records it as API-captured. When the request actually arrived by email
or phone, set `captured_via` to match the real channel — the dashboard
filters by it and DPB audits care about the provenance.

### SLA clock starts at `created_at`, not at identity verification

Even if you verified the requestor's identity two days before creating
the API record, the 30-day clock starts at the server's
`created_at` timestamp. Minimise intake-to-API latency.

### `type=nomination` is for DPDP §14 nominees

A `nomination` request is a data principal nominating another person
to exercise their rights if they become incapacitated or pass away.
Rare, but mandatory to support — and the fulfilment is *registering the
nominee*, not running deletions.

### Correction requests don't auto-update your data

The API doesn't know what "corrected" means for your schema. When
`type=correction`, capture the request, fulfil it in your own systems,
then close the rights_request row from the dashboard. The API is the
audit receipt, not the orchestrator.

### Don't trigger deletion for access requests

`type=access` means "give me a copy of my data" — a read-only operation.
Don't call `/v1/deletion/trigger` for these. Package the data (CSV /
PDF / JSON, whichever you're set up for) and deliver it via whatever
download link your legal team has blessed.

### Verify identity *before* the API call

The API accepts the request once you attest identity was verified. If
you haven't actually verified — and the DPB later disputes the
attestation — you're on the hook for a DPDP §13(4) violation. The
`identity_verified_by` string is a forensic claim.
