# Record consent at checkout

## Problem

The customer is on your checkout page, ticking a marketing-opt-in
checkbox along with the purchase. You need the consent recorded as an
immutable artefact for DPDP §6(4) — but you can't block the "Place
order" button on a third-party API call, and you can't lose the consent
if your server briefly can't reach ConsentShield.

## Shape

The call happens **server-side** right after your payment callback
confirms the order. Three rules:

1. **Non-blocking** — don't await the consent call inline with the order response.
2. **Idempotent** — use `client_request_id` so retries can't double-record.
3. **Durable** — on failure, persist the consent envelope and let a background worker retry.

<Callout tone="info" title="Why server-side, not browser">
Bearer tokens (`cs_live_*`) must never ship to the browser. The
consent checkbox lives in the form; the actual API call lives in your
order-confirmation handler. If you need browser-side capture, use the
ConsentShield banner script, which uses a short-lived HMAC signing
secret, not an API key.
</Callout>

## Prerequisites

- Purpose defined in the dashboard — grab its `purpose_definition_id` via `GET /v1/purposes`.
- Web property registered — grab its `property_id` via `GET /v1/properties`.
- API key with the `write:consent` scope.

```bash
# One-time setup: fetch the purpose + property IDs you'll hard-code.
curl -s "https://api.consentshield.in/v1/purposes" \
  -H "Authorization: Bearer $CS_API_KEY" | jq '.items[] | {id, purpose_code}'

curl -s "https://api.consentshield.in/v1/properties" \
  -H "Authorization: Bearer $CS_API_KEY" | jq '.items[] | {id, domain}'
```

export async function recordCheckoutConsent(job: CheckoutConsent) {

    data_principal_identifier: job.customerEmail,
    identifier_type:           'email',
    purpose_definition_ids:          job.acceptedPurposes,

    client_request_id: \`order-\${job.orderId}:checkout-consent\`,

      body: JSON.stringify(payload),
      signal: AbortSignal.timeout(5_000),

  )

  if (res.status === 201 || res.status === 200) {
    const body = await res.json()
    return { ok: true as const, event_id: body.event_id, replay: body.idempotent_replay }

  if (res.status >= 500 || res.status === 429) {
    // Retriable. Requeue with exponential backoff.
    throw new Error(\`retriable: \${res.status}\`)

  // 4xx → config drift. Log + surface; do NOT retry.
  const problem = await res.json()
  throw new Error(\`non-retriable: \${problem.title} — \${problem.detail}\`)
}

    rejectedPurposes: order.marketingOptIn ? [] : [process.env.CS_PURPOSE_MARKETING!],
  })
  return { orderId: order.id, status: 'confirmed' }
}`,

## Gotchas

### `captured_at` drift

The server rejects `captured_at` values more than **±15 minutes** from
its own clock. Use your app server's time, not the browser's. If your
background worker picks up a queued job hours after checkout, update
`captured_at` to the current wall-clock — the artefact's `created_at`
still records when the artefact was actually written, and a separate
`captured_via` field signals the original channel.

### Idempotency ≠ deduplication across orders

`client_request_id` makes the **same** call idempotent. Two different
orders that happen to use the same customer email must use different
`client_request_id` values (order-42 vs order-43) or the second call
will return the first order's artefact as a replay. Always namespace
with the order ID (or similar stable primary key).

### Record rejections too

`rejected_purpose_definition_ids` is what DPDP §11 calls the *audit
trail* for purposes the principal was shown but didn't accept. It
doesn't create artefacts, but it writes an immutable row to
`consent_events` that proves the choice was offered. Skip it and
you're missing half the compliance picture.

### Don't block on failure

If the consent API is temporarily unreachable, your order must still
complete. Two durable patterns:

- **Outbox pattern** — write the consent envelope to your own DB in the same transaction as the order, then have a background worker drain it. Failure never loses a consent record.
- **Queue retry** — push the job to SQS/RabbitMQ with visibility-timeout retries. Simpler, but relies on at-least-once delivery.

Either way, the `client_request_id` protects against duplicates.

### 410 Gone means someone rotated the key

Not a transient failure. See
[key rotation & tombstones](/docs/concepts/key-rotation-and-tombstones).
Page whoever rotates production keys, not oncall.

## Verify it worked

Right after a successful call, your user-facing dashboard can confirm:

```bash
curl "https://api.consentshield.in/v1/consent/verify/batch" \
  -X POST \
  -H "Authorization: Bearer $CS_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "property_id":     "3a4b8b21-5a11-4f37-9b7c-6f2e0d5c8b1a",
    "identifier_type": "email",
    "purpose_code":    "marketing",
    "identifiers":     ["alice@example.com"]
  }'

# { "results": [{ "identifier": "alice@example.com", "status": "granted", ... }] }
```
