# Build a preference center

## Problem

Your logged-in users want a page that shows every consent they've ever
given, which ones are currently active, and lets them flip any of them
off without filing a rights request. DPDP §6(5) makes withdrawal a
first-class operation — easier to revoke than to grant. Your preference
center has to mirror that reality.

## Shape

A preference center is three API calls:

1. **`GET /v1/consent/artefacts?data_principal_identifier=…&identifier_type=…&status=active`** — pull the user's active artefacts.
2. **Render one row per purpose** — the `purpose_code` + `data_scope` are the user-visible labels.
3. **On "Turn off"** — `POST /v1/consent/artefacts/{id}/revoke` with `reason_code=user_preference_change`.

For purposes the user *hasn't* granted yet, the preference center is a
**grant-new** flow — `POST /v1/consent/record` with the chosen
`purpose_definition_ids`, exactly like the checkout recipe.

<Callout tone="info" title="The artefact, not the event, is the UI model">
Users reason about *the consent*, not *the decision moment*. A preference
center lists **active artefacts** (one row per current permission), not
consent events. Events are the audit log behind it.
</Callout>

// GET /api/preferences → list the current user's active artefacts.
export async function GET(req: NextRequest) {
  const session = await getSession(req)             // your auth helper
  if (!session) return NextResponse.json({ error: 'unauthenticated' }, { status: 401 })

  const qs = new URLSearchParams({
    data_principal_identifier: session.email,
    identifier_type:           'email',
    status:                    'active',
    limit:                     '100',

      code:        a.purpose_code,
      scope:       a.data_scope,
      expires_at:  a.expires_at,
    })),
  })
}

## UI hints

- **Always show `expires_at`** when present — DPDP lets you specify a default purpose-expiry, and users appreciate transparency about re-consent schedules.
- **Show `data_scope`** as human-readable chips under each purpose. The list is maintained in the dashboard's purpose definitions; treat it as the source of truth.
- **Group by framework** when you run a mix — `dpdp`, `abdm`, `gdpr` in the artefact envelope signals which regulation is attached.
- **Never mutate in place.** A revoke-then-grant sequence is two artefacts with a chronological replacement chain, not an update. The preference center should reflect this: "Off since Apr 24" + "On again Apr 26" is the audit truth.

## Gotchas

### Listing by identifier needs both fields

`GET /v1/consent/artefacts` with `data_principal_identifier` requires
`identifier_type` in the same request. Passing one without the other
returns **422**. The server needs the type to normalise the identifier
consistently (email → lower-cased, phone → E.164, etc.).

### Large artefact histories — paginate

The endpoint is cursor-paginated; `limit` caps at 200. A user who has
been with you for 5 years may have 50+ artefacts across
revoke/grant cycles. Fetch the first page, render, and
prefetch subsequent pages on scroll if your UI needs the full history.
Treat `next_cursor: null` as the end marker.

### Revoke is the *only* way to turn off

There is no "update" on an active artefact. To switch a purpose off,
revoke it. If the user wants to turn it back on later, `POST
/v1/consent/record` creates a **new** artefact. The old artefact stays
in the record as revoked — DPDP §11 wants every decision durable.

### Revoking an already-revoked artefact

A second revoke on the same `artefact_id` returns **409 Conflict**.
Treat as idempotent from the UI's perspective — refresh the list and
move on. Don't retry automatically.

### Users want to see *why* a purpose expired

`status=expired` happens when `expires_at` passes without re-consent.
Render a distinct visual state for `expired` vs `revoked` — user-
triggered revocation and automatic expiry are very different events
from the user's mental model.
