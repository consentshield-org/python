# Error codes

The ConsentShield v1 API returns **RFC 7807 `application/problem+json`**
for every non-2xx response. The body carries a stable `type` URI,
human-readable `title`, HTTP `status`, and specific `detail`, plus
optional extensions. Your integration can branch on `status` for the
class of failure and surface `detail` for user-visible hints without
pattern-matching free text.

## Response shape

```json
{
  "type":    "https://consentshield.in/errors/unprocessable-entity",
  "title":   "Unprocessable Entity",
  "status":  422,
  "detail":  "purpose_codes must be an array of strings"
}
```

- `type` — stable URI. Slug form of `title`. Safe to switch on.
- `title` — HTTP reason phrase. Never changes for a given `status`.
- `status` — HTTP status code (echoes the response line).
- `detail` — specific, localised-to-code reason. Safe to render to end users but **may change wording** across releases.
- `instance` *(optional)* — request-scoped URI when the error is tied to a specific record.

## Authentication errors (401 / 410)

Emitted by the auth middleware before your handler runs. Do not rely on
the `detail` wording to distinguish cases; branch on `status`.

  },
  {
    name: '410 · revoked',

<Callout tone="tip" title="Treat 410 as a routine deploy signal">
After rotation, the old key returns **410 Gone** for two years. This is a
deliberate design choice so client libraries can emit a clear rotation
alert instead of a generic 401. Your monitoring should log 410s at
*info* level, not *error*.
</Callout>

## Server errors (500)

`500` means something on our side failed unexpectedly. Safe to retry
with exponential backoff. Please open a support ticket with the
`X-Request-Id` value if it persists — we log every 500 with that ID.

Common `detail` values (each corresponds to the endpoint that failed):

- `Audit listing failed`
- `Consent record failed`
- `Deletion trigger failed`
- `Key introspection failed`
- `Plan listing failed`
- `Property listing failed`
- `Purpose listing failed`
- `Revocation failed`
- `Rights-request creation failed`
- `Rights-request listing failed`
- `Score read failed`
- `Security-scans listing failed`
- `Usage lookup failed`
- `Verification failed`
- `test_delete failed`
- `List failed`, `Lookup failed`, `Update failed`

## Retry policy

| Status | Retriable? | Strategy |
|--------|-----------|----------|
| 400, 401, 403, 404, 409 (except concurrent-update), 410, 413, 422, 501 | No | Fix the request; do not retry. |
| 409 (concurrent update) | Yes | Re-fetch current state; retry once. |
| 429 | Yes | Honour `Retry-After`. Add ±10% jitter. |
| 500, 502, 503, 504 | Yes | Exponential backoff: 1s, 2s, 4s, 8s, capped at 60s. Max 5 attempts. Include `X-Request-Id` on each attempt. |

Client libraries (ADR-1006) ship this policy by default; the cURL and
raw-`fetch` snippets in [rate limits](/docs/rate-limits) show how to
reproduce it in-house.

## Opening a support ticket

When you contact support:

1. Include the full `X-Request-Id` header value from the failing response.
2. Include the raw response body (status + headers + body) — the `type` + `detail` tell us the code path without exposing any tenant data.
3. Do **not** include the `Authorization` header or the plaintext API key — our support channel is not rated for secret material.
