# SDK availability

Six official client libraries across two tiers. They share one
compliance contract, one error model, and one fail-CLOSED default —
pick the one that matches your stack.

## Tier 1 — hand-rolled

Line-audited, exact-pinned, written by us against the v1 API directly.

| Language | Package | Install | Min version |
|---|---|---|---|
| Node.js / TypeScript | [`@consentshield/node`](/docs/sdks/node) | `npm install @consentshield/node` | Node 18+ |
| Python | [`consentshield`](/docs/sdks/python) | `pip install consentshield` | Python 3.9+ |
| Go | [`github.com/SAnegondhi/consentshield-go`](/docs/sdks/go) | `go get github.com/SAnegondhi/consentshield-go` | Go 1.22+ |

## Tier 2 — generated from OpenAPI <span title="Generated from the public OpenAPI spec at api.consentshield.in/openapi.yaml — same compliance contract, hand-written framework wrappers" style={{fontSize:'0.8em',opacity:0.7}}>(generated)</span>

OpenAPI-Generator output from the [public spec](https://api.consentshield.in/openapi.yaml), with hand-written framework wrappers (Spring Boot starter / ASP.NET Core DI / Laravel + Symfony) layering the same fail-CLOSED retry contract on top.

| Language | Package | Install | Min version |
|---|---|---|---|
| Java (Spring Boot) | [`consentshield-java-spring-boot-starter`](/docs/sdks/java) | Maven `com.consentshield:consentshield-java-spring-boot-starter:1.0.0` | JDK 11+ |
| .NET (ASP.NET Core) | [`ConsentShield.Client.AspNetCore`](/docs/sdks/dotnet) | `dotnet add package ConsentShield.Client.AspNetCore` | .NET 8.0 LTS |
| PHP (Laravel / Symfony) | [`consentshield/sdk`](/docs/sdks/php) | `composer require consentshield/sdk` | PHP 8.1+ |

All six are at **v1.0.0**, **licensed under Apache 2.0** — fork, embed, ship in commercial products, no per-seat fees on the SDK itself. The "ConsentShield" name and logo are reserved as trademarks; see the `NOTICE` file in each repo for the carve-out.

<Callout tone="info" title="Why hand-rolled vs generated?">
Tier 1 (Node / Python / Go) carries the compliance audit load — we
write each SDK by hand, audit it line-by-line, and pin the one
absolute minimum dep (httpx in Python, tsup build-only in Node, zero
runtime deps in Go). Tier 2 (Java / .NET / PHP) is OpenAPI-Generator
output — backend integrators expect generated SDKs (every major API
vendor ships them that way), and the compliance-load-bearing parts
(retry middleware, fail-CLOSED, exception hierarchy) live in the
hand-written wrapper layer that's identical regardless of generator.
Mobile (Swift / Kotlin) is a separate ADR — different security model
entirely.
</Callout>

## What every SDK does

The same things, in the language's idiom:

- **`Verify` / `VerifyBatch`** — single + bulk consent checks
  (compliance-load-bearing).
- **`RecordConsent`** — capture a consent event, idempotent via
  `client_request_id`.
- **`RevokeArtefact`** — server-initiated revocation; 409 surfaces
  on terminal-state.
- **`ListArtefacts` / `IterateArtefacts` / `GetArtefact`** —
  cursor-paginated reads (returns `null` for unknown ids).
- **`ListEvents` / `IterateEvents`** — consent event stream.
- **`TriggerDeletion` / `ListDeletionReceipts`** — deletion fan-out
  + receipt walk.
- **`CreateRightsRequest` / `ListRightsRequests`** — DPDP §13/§14
  rights flow.
- **`ListAuditLog` / `IterateAuditLog`** — audit replay.
- **`Ping`** — `/v1/_ping` health check.

## The compliance contract — every SDK, identical

| Outcome | Default (fail-CLOSED) | `failOpen=true` |
|---|---|---|
| 4xx (caller bug / scope / 422 / 404 / 413) | error / exception | error / exception (4xx **always** surfaces) |
| Timeout | error / exception | open envelope, `cause: 'timeout'` |
| Network (DNS / TCP / TLS reset) | error / exception | open envelope, `cause: 'network'` |
| 5xx | error / exception | open envelope, `cause: 'server_error'` |

The default — fail-CLOSED — is the safe one. When ConsentShield is
briefly unreachable, your service responds **503** to its own caller
rather than silently default-granting consent that may have been
withdrawn 30 seconds ago.

To opt into fail-OPEN per call site, set `failOpen` (Node), `fail_open`
(Python), or `FailOpen` (Go) — or the env var
`CONSENT_VERIFY_FAIL_OPEN=true` for service-wide opt-in. Wire your
audit-trail callback (`onFailOpen` / `on_fail_open`) so every override
lands on Sentry / your structured logger / a custom audit POST.

## Framework-integration examples ship in every SDK

| SDK | Examples |
|---|---|
| Node.js | Express middleware, Next.js App Router |
| Python | Django middleware, Flask decorator, FastAPI dependency |
| Go | net/http (chi-compatible), gin |
| Java | Spring Boot 3 marketing gate (auto-configured `ConsentShieldClient` bean) |
| .NET | ASP.NET Core 8 minimal-API marketing gate (`AddConsentShield(IConfiguration)`) |
| PHP | Laravel 11 middleware, Symfony 7 controller |

Each example refuses with HTTP 451 on `revoked` / `expired` /
`never_consented`, fails CLOSED with HTTP 503 on a 5xx / network /
timeout, and surfaces `X-CS-Trace-Id` for end-to-end correlation.

</Callout>
