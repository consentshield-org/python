# `@consentshield/node`

The official TypeScript-first Node.js client. Hand-rolled, exact-pinned,
zero npm runtime deps beyond the build toolchain.

## Install

```bash
npm install @consentshield/node
# or
pnpm add @consentshield/node
# or
yarn add @consentshield/node
# or
bun add @consentshield/node
```

- **Node 18+** required
- Ships dual **ESM (`.mjs`)** + **CJS (`.cjs`)** + `.d.ts` / `.d.cts`
- `sideEffects: false` â€” tree-shake friendly
- Typed top-to-bottom (no `any` in the public API)

## Quickstart

```ts
import { ConsentShieldClient } from '@consentshield/node'

const client = new ConsentShieldClient({
  apiKey: process.env.CS_API_KEY!,
})

const result = await client.verify({
  propertyId: 'PROP_UUID',
  dataPrincipalIdentifier: 'user@example.com',
  identifierType: 'email',
  purposeCode: 'marketing',
})
if (result.status !== 'granted') {
  // 451 your caller
}
```

## Configuration

```ts
new ConsentShieldClient({
  apiKey: 'cs_live_...',     // required, validated at construction
  baseUrl: 'https://api.consentshield.in', // default
  timeoutMs: 2000,           // per-attempt; never retried on timeout
  maxRetries: 2,             // 5xx + transport errors only
  failOpen: false,           // see compliance contract below
  onFailOpen: (env, ctx) => { /* audit trail */ },
})
```

The `CONSENT_VERIFY_FAIL_OPEN=true` env var flips `failOpen` for
service-wide opt-in. Explicit constructor option always wins.

## Compliance contract

| Outcome | Default | `failOpen=true` |
|---|---|---|
| 4xx (caller bug / scope / 422 / 404 / 413) | throws `ConsentShieldApiError` | throws `ConsentShieldApiError` (4xx **always** throws) |
| Timeout | throws `ConsentVerifyError` wrapping `ConsentShieldTimeoutError` | returns `OpenFailureEnvelope`, `cause: 'timeout'` |
| Network | throws `ConsentVerifyError` wrapping `ConsentShieldNetworkError` | returns `OpenFailureEnvelope`, `cause: 'network'` |
| 5xx | throws `ConsentVerifyError` wrapping `ConsentShieldApiError` | returns `OpenFailureEnvelope`, `cause: 'server_error'` |

Use `isOpenFailure(result)` as an ergonomic type guard:

```ts
const result = await client.verify({ ...args })
if (isOpenFailure(result)) {
  // failOpen path â€” audit-trail wired via `onFailOpen`
  return next()
}
if (result.status !== 'granted') return res.status(451).end()
```

## Audit-trail wiring (`onFailOpen`)

Every fail-OPEN override fires the `onFailOpen` callback exactly
once. Wire it to your structured logger / Sentry / a custom audit
POST. The default emits `console.warn` with the envelope + method.

```ts
new ConsentShieldClient({
  apiKey: process.env.CS_API_KEY!,
  failOpen: true,
  onFailOpen: (envelope, ctx) => {
    Sentry.captureMessage('consent.fail_open', {
      level: 'warning',
      tags: { method: ctx.method, cause: envelope.cause },
      extra: { envelope },
    })
  },
})
```

The promise return is **not awaited** â€” the callback is fire-and-
forget. Throws inside the callback are caught and emitted via
`console.error`; they NEVER break the verify call site.

## Trace-id round-trip

```ts
const result = await client.verify({
  propertyId: 'PROP_UUID',
  dataPrincipalIdentifier: 'user@example.com',
  identifierType: 'email',
  purposeCode: 'marketing',
  traceId: req.headers['x-trace-id'] as string | undefined,
})
// result.traceId is the server-emitted trace id (echoes the request
// trace id when supplied; otherwise the server mints one).
```

Errors carry the trace id too:

```ts
catch (err) {
  if (err instanceof ConsentShieldError && err.traceId) {
    Sentry.setTag('cs.trace_id', err.traceId)
  }
  throw err
}
```

## Examples

The package ships two runnable integrations:

- **`examples/express-verify-middleware/`** â€” Express 5 middleware
  refusing with HTTP 451 on non-granted, HTTP 503 on fail-CLOSED.
- **`examples/nextjs-mobile-record/`** â€” Next.js 16 App Router API
  route with module-load init + structured error surface.

Both are at the package root after `npm install` â€” copy or
reference verbatim.
