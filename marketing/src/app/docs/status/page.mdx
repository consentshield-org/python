# Status & uptime

Live platform health lives at **[status.consentshield.in](https://status.consentshield.in)**.
It's hosted outside our primary infrastructure so the page stays
reachable when the API itself is degraded.

<Callout tone="info" title="Why the status page is external">
Embedding live status inside the docs site would couple uptime of the
status surface to uptime of the marketing / docs infrastructure. A
dedicated, independently-hosted status page is how every other API
platform worth trusting operates — and we follow the same rule.
</Callout>

## Incident notifications

- **Subscribe** — email / SMS / Slack / webhook on [status.consentshield.in](https://status.consentshield.in).
- **Incident severities** — `investigating` → `identified` → `monitoring` → `resolved`. Post-mortems for SEV-1 incidents land in [Resolved](https://status.consentshield.in/history) within 5 business days.
- **Enterprise plans** — dedicated Slack channel with direct oncall escalation.

## Historical uptime

The status page's [History](https://status.consentshield.in/history)
tab holds rolling 90-day incident logs with per-component availability
numbers. Older data is archived quarterly and available on request.

## Reporting an incident

If you're seeing symptoms the status page hasn't flagged yet:

1. **Check your own side first** — timeouts, DNS, firewall changes, clock drift.
2. **Retry with a new `X-Request-Id`** and capture the full response (status + headers + body).
3. **Email support@consentshield.in** with the `X-Request-Id`, the endpoint, and the time window. We log every 5xx with that ID.
4. **For confirmed platform incidents**, we update the status page within 15 minutes of detection.
