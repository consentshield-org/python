Quickstart — record your first consent artefact

In 15 minutes you will issue an API key, call POST /v1/consent/record, and verify the resulting artefact. If you get stuck, email dev-support — we reply within one business day.

TIME ≈ 15 min PREREQ ConsentShield account (sign up) DIFFICULTY Beginner
1

Issue an API key

In the dashboard go to Settings → API keys → New key. Give it a name (e.g. staging-integration) and copy the plaintext. You will only see it once — store it in a secret manager, never in source control.

The key looks like cs_live_01HRG…N7QZ. Keys are scoped to one organisation; a separate org_id is embedded inside.

2

Test the key

Call the health endpoint to confirm authentication works.

# Expect HTTP 200 + {"ok": true} curl "https://api.consentshield.in/v1/_ping" \ -H "Authorization: Bearer $CS_API_KEY"
3

Record a consent

The minimum viable request needs a data-principal identifier, a purpose code (fetched via GET /v1/purposes), and the consent outcome.

curl -X POST "https://api.consentshield.in/v1/consent/record" \ -H "Authorization: Bearer $CS_API_KEY" \ -H "Content-Type: application/json" \ -d '{ "data_principal": { "email": "a@example.in" }, "purpose_code": "marketing_email", "outcome": "granted", "origin": "https://shop.example.in/checkout" }'
4

Verify & archive

The response contains an artefact id. Use it to re-verify consent on every subsequent transaction, and to produce DPB-facing audit exports.

✓ You're done

That artefact is now the system of record. It survives banner changes, session drops, and employee churn. To invalidate it, call POST /v1/consent/artefacts/{id}/revoke — never delete rows directly.

Record consent at checkout

Capture marketing, analytics, and third-party-sharing consents at the point the data principal completes a purchase — without blocking the checkout submit.

TIME ≈ 20 min STACK Node.js · any frontend VERTICAL E-commerce · BFSI · SaaS

Problem

Your checkout already submits a payment to Razorpay / Stripe. You need to record which marketing + analytics consents the customer granted — and be able to prove it later, with a timestamp the DPB will accept and a signed artefact that survives banner changes.

The shape

  1. Frontend: the checkout form already has the consent checkboxes. On submit, stage the consent flags locally.
  2. Backend: once the payment succeeds, fire POST /v1/consent/record for each granted purpose, in parallel with the order-confirmation email.
  3. Non-blocking: the consent call never gates the checkout response. Retry with exponential backoff on 5xx.
⚠ Security

Never send the raw API key to the browser. The record-consent call is a server-side operation. Use /v1/consent/verify from the browser only if you need real-time banner-state queries — and even then prefer an edge proxy.

Full example

// checkout.ts — after Razorpay payment.success webhook import { fetch } from "undici" const CS_KEY = process.env.CS_API_KEY export async function recordCheckoutConsents(order: Order, consents: ConsentFlags) { const calls = Object.entries(consents) .filter(([_, v]) => v === true) .map(([purpose_code]) => fetch( "https://api.consentshield.in/v1/consent/record", { method: "POST", headers: { "Authorization": `Bearer ${CS_KEY}`, "Content-Type": "application/json", "Idempotency-Key": `${order.id}:${purpose_code}` }, body: JSON.stringify({ data_principal: { email: order.customer_email }, purpose_code, outcome: "granted", origin: order.checkout_url, occurred_at: order.paid_at }) } )) // Non-blocking — never fail checkout if ConsentShield is down. await Promise.allSettled(calls) }
◉ Idempotency

Pass an Idempotency-Key header. If the request is retried after a network blip, ConsentShield returns the original 201 artefact rather than creating a duplicate. Keys are scoped to your API key and live for 24 hours.

What the artefact looks like

{ "artefact_id": "cs_art_01HRG4FN7QZ3K8M2YD9XB5VJCT", "status": "active", "issued_at": "2026-04-21T14:02:11Z", "purpose": { "code": "marketing_email", "display_name": "Marketing emails about new products", "legal_basis": "consent" }, "receipt_url": "https://api.consentshield.in/v1/consent/artefacts/cs_art_01HRG…" }

Gotchas

  • Timezone. Always send occurred_at in UTC. The DPB expects IST but we convert at render-time — storing IST will double-shift.
  • Partial consents. If a customer grants marketing but declines analytics, record each purpose as its own row. Do not bundle into one artefact with a purposes[] array — that shape is reserved for DEPA bundled artefacts (see artefacts vs events).
  • Data principal identity. For logged-in users prefer data_principal.user_id. For guest checkouts, email is acceptable but will be hashed on the wire.

Docs/API reference/Consent/Record

Record a consent

Create a first-class consent artefact from a data principal's affirmative action. Idempotent when Idempotency-Key is provided.

POST https://api.consentshield.in/v1/consent/record
AUTH Bearer cs_live_* IDEMPOTENT Yes (via header) RATE LIMIT Plan-scoped STABILITY GA CHANGELOG v1 — unchanged since 2026-04-15

Request body

data_principal.email
string
Data-principal email, used as the hashed identifier in the artefact. Exactly one of email or user_id is required. REQUIRED*
data_principal.user_id
string
Your opaque user id (preferred for logged-in flows). Hashed at rest.
purpose_code
string
A purpose registered on your organisation — list via GET /v1/purposes. Case-sensitive. REQUIRED
outcome
"granted" | "denied" | "withdrawn"
The consent decision. withdrawn is reserved for prior consents being revoked — use POST /artefacts/{id}/revoke for the full workflow. REQUIRED
origin
string (URL)
Where the consent was captured. Must be a registered web property on your organisation, or the call is rejected 403.
occurred_at
string (ISO-8601 UTC)
When the consent was captured. Defaults to server receipt time. DEFAULT: now()
metadata
object ≤ 1 kB
Free-form non-PII tags. Stored unencrypted; do not put personal data here.

Response

201
Artefact created. Body is the full artefact object (see below).
200
Idempotency-key replay. Original artefact returned unchanged.
400
invalid_request — validation failure. error.field names the offending property.
401
unauthorised — missing, malformed, or revoked API key. 410 Gone instead if the plaintext was rotated.
403
origin_not_registered — the origin URL is not on the org's allowed_origins list.
404
purpose_not_found — purpose_code does not exist on this org.
429
Rate limit exceeded. Retry-After header present.
503
buffer_unavailable — retriable. Use exponential backoff.

Example

POST /v1/consent/record Authorization: Bearer cs_live_01HRG…N7QZ Content-Type: application/json Idempotency-Key: order-42:marketing_email { "data_principal": { "email": "a@example.in" }, "purpose_code": "marketing_email", "outcome": "granted", "origin": "https://shop.example.in/checkout", "occurred_at": "2026-04-21T14:02:11Z" }
