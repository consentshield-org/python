## Key Learnings

- **Next.js 16 uses `proxy.ts` not `middleware.ts`.** The proxy runs on Node.js runtime (not Edge). Export `proxy` not `middleware`. See docs/architecture/nextjs-16-reference.md.
- **All request APIs must be awaited in Next 16.** `await cookies()`, `await headers()`, `await params`, `await searchParams`. Sync access removed entirely.
- **`next lint` does not exist in Next 16.** Run `eslint src/` directly. The package.json script already accounts for this.
- **Caching is opt-in in Next 16.** Pages are dynamic by default. Use `"use cache"` directive only for static content. Good for ConsentShield's real-time buffer data views.
- **Tailwind v4 is CSS-first.** No `tailwind.config.js`. Uses `@theme` in CSS and `@tailwindcss/postcss`.
- **ESLint 9 flat config.** Uses `eslint.config.mjs`, not `.eslintrc.json`.
- **Every package must be latest with security patches, exact-pinned.** No `^`, no `~`. Check versions before adding any dependency.
- **Supabase hosted: `gen_random_bytes()` lives in `extensions` schema.** Must qualify as `extensions.gen_random_bytes()` in all migrations. Same applies to other pgcrypto functions.
- **Use pooler connection string for psql queries.** Direct DB hostname doesn't resolve from local. Use `postgresql://postgres.xlqiakmkdjycfiioslgs:PASSWORD@aws-1-ap-northeast-1.pooler.supabase.com:5432/postgres`.
- **Supabase REST accepts JWTs signed with the project JWT secret for ANY role claim — but the signing secret is being phased out.** Historically the Cloudflare Worker's `SUPABASE_WORKER_KEY` was an HS256 JWT claiming `role: cs_worker`, signed with the project's HS256 shared secret. PostgREST `SET ROLE`s based on the claim. That's why cs_worker ever "worked" with Supabase REST.
- **[2026-04-21] Supabase is rotating HS256 shared-secret → ECC P-256 asymmetric JWT signing.** The dashboard now lists the HS256 key as "Previously used (legacy)". Once revoked, every HS256-signed scoped-role JWT stops working — including SUPABASE_WORKER_KEY. ECC P-256 is Supabase-private (asymmetric), so we can NO LONGER mint new role JWTs from our side.
- **[2026-04-21] For scoped-role access going forward, use direct Postgres via the Supavisor pooler.** Grant the role `LOGIN` + a password; connect via `postgres.js` (app runtime) or `psql` (ops) against `aws-1-<region>.pooler.supabase.com:6543` (transaction mode for apps) or `:5432` (session mode for DDL). ADR-1009 Phase 2 established this pattern for cs_api; it's also the Worker's eventual migration path.
- **[2026-04-21] `sb_secret_*` (new Supabase API key format) is NOT the JWT signing secret.** It's an opaque service-role-equivalent token. You cannot sign JWTs with it. For signing you need the HS256 legacy secret (on its way out) or direct-Postgres access.
- **React 19 purity rule flags `Date.now()` and `new Date()` inline in server components.** Extract to helper functions in `lib/` (e.g., `nowIso()`, `isoSinceHours(n)`). The lint rule only catches inline calls, not those wrapped in named functions. Same applies to `Math.random` and other impure operations.
- **[2026-04-24] `bypassrls` on a Postgres role is NOT a substitute for table grants.** cs_orchestrator has bypassrls (so it sees every row across RLS policies), but SQL-level privilege checks are separate: a SELECT/INSERT/UPDATE/DELETE still needs the explicit grant. Found during ADR-1025 Sprint 2.1 E2E — cs_orchestrator hit "permission denied for table export_configurations" despite bypassrls. Added explicit grants in migration 20260804000037. **Audit stance going forward:** join `information_schema.role_table_grants` against cs_orchestrator to assert least-privilege. Treat bypassrls as a carveout for RLS-enabled tables, not a blanket allow. This applies to cs_api, cs_worker, cs_delivery too.
- **[2026-04-24] CF R2 S3 credentials are minted at user-level, not account-level.** `POST /user/tokens` (user bearer `cfut_*`) returns `{id, value, …}`. S3 `access_key_id = token.id`, `secret_access_key = sha256hex(token.value)` (hex, not base64). `/accounts/{id}/tokens` rejects R2 policies with 9109 "Valid user-level authentication not found" regardless of scopes — this is a platform constraint, not a scope issue. Resource key format: `com.cloudflare.edge.r2.bucket.<account>_default_<bucket>`. Permission group UUIDs: Read `6a018a9f2fc74eb6b293b0c548f38b39`, Write `2efd5506f9c8494dacb1fa10a3e7d5b6` (covers read+write+delete). **Token + revocation both have ~5-10s propagation delay** at the R2 edge — production flows must account for this.
- **[2026-04-24] CF Workers free-tier subrequest limit is 50 per invocation; paid tier is 10 000.** Each network round-trip from a Worker to a non-CF origin counts — DNS, TCP connect, TLS handshake stages, SCRAM auth exchanges, each query packet. A cold `postgres.js` connection to Supabase Supavisor (`aws-1-ap-northeast-1.pooler.supabase.com:6543`) exceeds 50 subrequests in the handshake alone, before any SELECT. `max: 1` doesn't help — budget is consumed in the handshake, not the pool size. **Hyperdrive's internal `*.hyperdrive.local` hostname does NOT count against this budget** — that's the actual load-bearing value of Hyperdrive for ConsentShield's Worker on the free plan, not edge-pool latency or query-result caching (the caching path is disabled because Supavisor transaction mode can't reuse prepared statements). Configurable via `limits` in `wrangler.toml`, but the free-tier ceiling is account-plan-level. ADR-1026 was abandoned on this constraint. Reopen if Workers Paid is ever adopted.
- **[2026-04-24] CF API tokens need per-service permissions explicitly.** A token that works for `wrangler deploy` + `wrangler secret put` will still return `code: 10000` (Authentication error) for `wrangler hyperdrive *` commands unless the token has `Account › Hyperdrive › Edit` checked. Narrowly-scoped tokens break silently for services added later — add permissions incrementally as the stack grows. Check permissions at https://dash.cloudflare.com/profile/api-tokens, edit in place — no token rotation needed.
- **[2026-04-24] Hyperdrive upstream pool can enter a stuck half-open state.** Caused by bursts of failed-auth / CONNECTION_ENDED / client-abandoned sockets accumulating. Symptoms: SQLSTATE 58000 "Timed out while waiting for an open slot in the pool" on every query, `pg_stat_activity` shows zero sessions from the user, direct psql to the same Supavisor endpoint works. `wrangler hyperdrive update` (even with the same password) does NOT reset the pool. Recovery: `wrangler hyperdrive delete` + `wrangler hyperdrive create` for a fresh binding ID; update `wrangler.toml` + redeploy. Supavisor additionally rate-limits CF's control-plane IPs after the burst, so CF dashboard pre-save validation can reject the new config even when the runtime path would succeed — use `wrangler hyperdrive update --origin-password` from CLI, which skips the pre-save probe.

[2026-04-21] **Don't `source .secrets` naively.** Values with a trailing `\` (line-continuation) join with the next line when bash sources the file. Example: `SUPABASE_DATABASE_PASSWORD=jxFENChEAG4cZdjZ\` plus the next line produced a 77-char mangled password instead of the 16-char real one, causing psql auth failures. Parse individual values with `grep "^KEY=" .secrets | sed 's/^KEY=//; s/\\$//'` or load into a variable via `read` with appropriate quoting. Also relevant to other scripts/tools that read .secrets.

[2026-04-21] **Don't assume `sb_secret_*` is the JWT signing secret.** It's an opaque API key, not a signing secret. Using it with HS256 produces a JWT that Supabase won't verify. The JWT signing secret lives in Project Settings → JWT Keys → Legacy HS256 (on new projects it may already be absent; those are direct-Postgres-only for scoped roles).
